# AGENTS.md — BuildWarden Developer Context

## What This Project Is

BuildWarden is a build-time network auditing platform. It orchestrates two containers on an isolated network:

- **Relay** (`10.0.87.2`) — An ephemeral HTTPS MITM proxy that intercepts all build traffic, records it to a cryptographically-signed ledger, and serves as DNS resolver for the build container. Also handles artifact ingestion via `POST http://artifacts/<name>`.
- **Build Container** (`10.0.87.3`) — A Docker-in-Docker environment where the actual build runs. Network-isolated via iptables to only communicate with the relay.

The output is a tamper-evident ledger of all network I/O during the build.

## Key Design Decisions

### Cryptographic Architecture (Two-Key Design)
- **MITM CA** — RSA-2048, generated by the relay at startup. Used solely for TLS interception. Private key NEVER leaves the relay container's memory. Public cert exported to `/ledger/ca.cert.pem` for the orchestrator to inject into the build container.
- **Ledger Signing** — Ed25519, generated by the relay at startup. Used for signing ledger entries (fast: ~50k signs/sec). Public key in `ledger.cert.pem`.
- **Binding** — The ledger header includes `ca_fingerprint` (SHA-256 of the MITM CA cert DER), creating a verifiable link without sharing keys.

### Orchestrator ↔ Relay Flow
1. Orchestrator creates ledger temp dir, builds relay image, creates network
2. Orchestrator starts relay container (ledger dir mounted at `/ledger`)
3. Relay generates ephemeral CA, writes `ca.cert.pem` to `/ledger/`
4. Orchestrator polls for `ca.cert.pem` (up to 5s)
5. Orchestrator runs extensions (inject CA into build container trust store)
6. Orchestrator starts build container, configures iptables, starts dockerd
7. Build runs with all traffic flowing through relay

### Streaming Hashes (Performance Critical)
Response bodies are hashed incrementally via `io.MultiWriter` over 4 hash algorithms (blake2b_256, sha256, sha1, md5) as bytes stream through `Read()`. This avoids buffering multi-GB payloads in memory. The `CheckpointHashed`/`CloseHashed` methods accept pre-computed hashes to skip redundant computation in the serialized ledger loop.

### Artifact Endpoint
- DNS resolves `artifacts` hostname to relay IP (10.0.87.2)
- `POST http://artifacts/<name>` is intercepted by `handleArtifactPost()`
- Body streamed to `payloads/<sha256>`, symlink created at `artifacts/<name>`
- Recorded in ledger with schema "artifact"
- NOT forwarded externally

### Reserved DNS Hostnames
Defined in `reservedHosts` map in relay.go. Currently: `artifacts`. These resolve to the relay's own IP instead of being forwarded to external DNS.

## Ledger Format (v2)
- Newline-delimited JSON
- First entry: header (version, hashes, environment, ca_fingerprint, signature_scheme)
- Subsequent: open/checkpoint/close entries with chained Ed25519 signatures
- Each entry signs over: previous signature + entry type + direction + payload hashes
- Payload identity: size + 4 hashes (blake2b_256, sha256, sha1, md5)

## Extension System
Extensions implement `BeforeBuild(env *CtrEnv) error` and optionally `Env() map[string]string`. They run on the host before the build starts and can write files to `.warden/` (which gets COPY'd into the build image) and scripts to `.warden/ext.d/` (which get exec'd inside the build container).

## Container Runtime Abstraction
Uses `github.com/lesiw/ctrctl` which wraps docker/finch/podman CLI. The `ctrctl.Verbose = true` flag prints all container commands with `+` prefix (useful for debugging).

## Finch VM Configuration
For heavy builds (TensorFlow), the finch VM needs:
```yaml
# ~/.finch/finch.yaml
cpus: 6
memory: 16GiB
datadisk: 150GiB
```

## Watchdog Script
`watchdog-build.sh` wraps the warden build with:
- Stall detection (default 600s without log output)
- Max timeout (default 4 hours)
- Automatic cleanup of containers/networks on timeout

## Known Issues / TODOs
- `editContainerfile()` modifies the original Containerfile in-place (has a FIXME to use a copy in .warden/)
- The `docker:dind` build container needs `SSL_CERT_FILE` env passed to dockerd for cert trust
- TF build requires `PIP_DEFAULT_TIMEOUT=300` in the Dockerfile ENV for Bazel's hermetic pip
- `--config=nogcp` needed for TF 2.21 to avoid downloading google_cloud_cpp (saves disk)
- `--config=noaws` and `--config=nohdfs` do NOT exist in TF 2.21 (removed in newer versions)
- The relay's `Dockerfile.relay` in the repo root is unused; the relay image is built dynamically by the orchestrator
