# AGENTS.md — BuildWarden Developer Context

## What This Project Is

BuildWarden is a build-time network auditing platform. It orchestrates two containers on an isolated network:

- **Relay** (`10.0.87.2`) — An ephemeral HTTPS MITM proxy that intercepts all build traffic, records it to a cryptographically-signed ledger, and serves as DNS resolver for the build container. Also handles artifact ingestion via `POST http://artifacts/<name>`.
- **Build Container** (`10.0.87.3`) — A rootless Docker-in-Docker environment where the actual build runs. Network-isolated via iptables + route replacement to only communicate with the relay.

The output is a tamper-evident binary ledger of all network I/O during the build.

## Running

```sh
# Autodetects runtime (finch > docker > podman)
./warden build examples/Dockerfile.simple

# Explicit runtime
./warden build --runtime docker examples/Dockerfile.simple

# Interactive shell in build environment
./warden shell ./examples
```

Configuration via `warden.toml` or `~/.config/warden/config.toml`.

## Key Design Decisions

### Cryptographic Architecture (Two-Key Design)
- **MITM CA** — RSA-2048, generated by the relay at startup. Used solely for TLS interception. Private key NEVER leaves the relay container's memory.
- **Ledger Signing** — Ed25519, generated by the relay at startup. Used for signing ledger entries (~50k signs/sec). Public key embedded in ledger header.

### Orchestrator ↔ Relay Flow
1. Orchestrator creates ledger temp dir, builds relay image, creates network
2. Orchestrator starts relay container (ledger dir mounted at `/ledger`)
3. Relay generates ephemeral CA, writes `ca.cert.pem` to `/ledger/`
4. Orchestrator polls for `ca.cert.pem` (up to 5s)
5. Orchestrator runs extensions (inject CA into build container trust store)
6. Orchestrator starts build container, configures iptables, starts dockerd
7. Build runs with all traffic flowing through relay

### Binary Ledger Format
- Magic: `BLDL` + version byte (0x01)
- Self-describing header: signature scheme, sizes, public key
- Record types: open, checkpoint, close, artifact
- Payload direction encoded in signed int64 (positive=in, negative=out)
- CBOR metadata (not signed, freely redactable)

### Artifact Endpoint
- DNS resolves `artifacts` hostname to relay IP (10.0.87.2)
- `POST http://artifacts/<name>` is intercepted by the relay
- Body streamed to `payloads/<sha256>`, symlink at `artifacts/<name>`
- Recorded in ledger with dedicated `artifact` record type

## Extension System
Extensions implement `BeforeBuild(env *CtrEnv) error` and optionally `Env() map[string]string`. They run on the host before the build starts and can write files to `.warden/` (COPY'd into build image) and scripts to `.warden/ext.d/` (exec'd inside the build container).

## Container Runtime Abstraction
Uses `github.com/lesiw/ctrctl` which wraps docker/finch/podman CLI. Runtime is autodetected or configured via `warden.toml` / `--runtime` flag.

## Finch VM Configuration
For heavy builds (PyTorch), the finch VM needs:
```yaml
# ~/.finch/finch.yaml
cpus: 6
memory: 16GiB
datadisk: 150GiB
```
